# Bundle Overview

## lifecycle-methodology

Purpose:
- route architecture work to the correct lifecycle stage
- keep stage boundaries explicit
- preserve survey -> solution design -> technical overview -> technical detailed design -> deployment/ops handoff as one coherent delivery chain

Skills:
- `arch-lifecycle-delivery`
- `arch-lifecycle-demand-survey-methodology`
- `arch-lifecycle-solution-design-methodology`
- `arch-lifecycle-tech-overview-methodology`
- `arch-lifecycle-tech-detailed-methodology`
- `arch-lifecycle-deploy-ops-methodology`

Best when:
- teams skip survey
- the current stage is unclear
- architecture artifacts are produced in the wrong order

## diagramming

Purpose:
- keep different architecture diagrams responsible for different questions
- reduce ambiguity in business flow, domain, system, technical, and deployment views

Skills:
- `arch-lifecycle-solution-design-biz-arch-diagramming`
- `arch-lifecycle-solution-design-biz-flow-diagramming`
- `arch-lifecycle-solution-design-domain-modeling`
- `arch-lifecycle-tech-overview-functional-arch-diagramming`
- `arch-lifecycle-tech-overview-system-arch-diagramming`
- `arch-lifecycle-tech-detailed-technical-arch-diagramming`
- `arch-lifecycle-tech-detailed-core-flow-diagramming`
- `arch-lifecycle-deploy-ops-deployment-arch-diagramming`

Best when:
- teams make one picture try to explain everything
- system architecture and technical architecture keep getting confused
- reviewers ask for diagrams but the deliverable layer is unclear

## architecture-supplements

Purpose:
- sharpen cross-cutting architecture judgment that does not belong to only one lifecycle stage

Skills:
- `ddd-domain-modeling-for-architecture`
- `design-patterns-and-refactoring`
- `technology-adoption-comparison`

Best when:
- boundary modeling dominates the work
- design choices require explicit reuse/adoption judgment
- code-structure guidance must stay aligned with architecture decisions
